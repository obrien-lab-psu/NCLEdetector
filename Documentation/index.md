## What EntDetect is for

**EntDetect** is a Python package for studying **non-covalent lasso entanglements (NCLEs)** in proteins and protein ensembles. It supports four major use cases:

1. **Identifying native entanglements** in a single protein structure.
2. **Tracking changes in entanglement** across simulation trajectories and organizing those changes into structural states.
3. **Connecting simulated structural ensembles to experimental data**, especially high-throughput conformational readouts such as **LiP-MS** and **XL-MS**.
4. **Testing whether experimentally observed conformational changes are enriched in proteins or regions containing native entanglements** at the proteome or population level.

## Who this tutorial is for

This tutorial assumes you:

- are working in a **Linux environment**;
- have already installed the software dependencies listed in the EntDetect setup documentation;
- have created a **Conda environment** named `entdetect` (or equivalent) with EntDetect installed;
- are comfortable running terminal commands and basic Python scripts.

### 4. Clean your structural inputs first

For structure-based analyses, ensure your input structure has been preprocessed:

- rebuild missing residues and atoms (e.g., with **Modeller** or **CHARMM**);
- remove duplicate residues;
- verify chain labels and residue numbering;
- match the coordinate representation expected by the analysis step.

## Workflow overview

The full EntDetect analysis logic can be thought of as four connected layers:

| Layer | Workflow | Key steps |
|-------|----------|-----------|
| **A. Native structure** | Workflow 1 | Detect NCLEs → filter → cluster → compute features |
| **B. Trajectory** | Workflow 2 | Compute Q/G/K → cluster entanglement changes → build MSM → analyze states |
| **C. Sim-to-experiment** | Workflow 3 | Back-map CG → compute SASA/Jwalk/XP → consistency test vs LiP-MS / XL-MS |
| **D. Population/statistical** | Workflow 4 | Native NCLEs for proteome → logistic regression → Monte Carlo subpopulation ranking |

## Common pitfalls and practical advice

### 1. Path mismatches are the most common source of errors

Double-check every input path before running a long analysis — especially structure files, trajectory files, output directories, mapping files, and experimental spreadsheet paths.

### 2. Keep naming consistent

Use the same identifier (`ID`) consistently across files:

```python
ID  = "1ZMR"
Traj = 1
```

Filenames that share the same protein/trajectory number are far easier to debug.

### 3. Save scripts instead of relying only on interactive sessions

Interactive use is helpful for testing, but production analyses should be placed in version-controlled Python scripts.

### 4. Downsample when appropriate

Order-parameter calculation and structural ensemble analysis can be very slow on large trajectories. Downsample to reduce runtime, disk usage, and memory usage when the science permits.

### 5. Always visually inspect key structural results

Do not rely solely on tables and scores. Visual inspection is essential at several stages:

- native NCLE detection;
- mirror-artifact removal;
- non-native entanglement clustering;
- metastable-state interpretation;
- representative-structure validation.

Use **VMD** or **PyMOL** for these inspections.

### 6. Choose model parameters deliberately

Important tunable parameters:

| Parameter | Workflow | Description |
|-----------|----------|-------------|
| `g_threshold` | 1 | Native entanglement detection sensitivity |
| `start` | 2, 3 | First frame to include in trajectory analysis |
| `n_large_states` | 2 | Number of metastable states in MSM |
| Downsampling interval | 2, 3 | Frequency of frame analysis |
| LiP-MS / XL-MS significance thresholds | 3, 4 | Preprocessing stringency |

These are scientific choices, not just technical settings.

### 7. Monitor memory usage for clustering steps

Non-native entanglement clustering can require substantial RAM when the number of raw entanglement-change events is large. Check available memory before running.

### 8. Keep intermediate outputs

Many downstream steps depend on earlier preserved files:

- combined `G` metadata (`Combined_GE/`);
- clustering archives (`.npz`, `.pkl`);
- MSM mapping files (`.csv`);
- SASA and Jwalk outputs (`.npy`);
- processed experimental tables.

Do not delete these prematurely.

## Final checklist for a new user

Before starting a full run, confirm **yes** to all of the following:

- [ ] Is my Conda environment active? (`conda activate entdetect`)
- [ ] Are my structure files cleaned and consistent?
- [ ] Do I know whether I am working with all-atom or coarse-grained data?
- [ ] Are my trajectory and metadata files in the expected locations?
- [ ] Have I tested the workflow on a small example first?
- [ ] Do I know which outputs from one step are required by the next?
- [ ] Have I planned enough storage, memory, and runtime for the larger steps?

## Suggested first run for beginners

If you are completely new to EntDetect:

1. Run **Workflow 1** on the provided `1zmr_model_clean.pdb` structure.
2. Inspect the representative NCLEs visually in VMD or PyMOL.
3. Compute entanglement features.
4. Only then move on to the trajectory and experimental workflows.

This staged approach lets you learn the package in manageable steps without committing to multi-day compute runs.
